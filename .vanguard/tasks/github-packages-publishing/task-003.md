### File Content

Create `RELEASE_CHANNEL` with exactly this content (no newline):

### Valid Channel Values

| Channel | npm Tag | Version Format | Use Case |
|---------|---------|----------------|----------|
| `alpha` | `alpha` | `0.1.0-alpha.1` | Early development, breaking changes expected |
| `beta` | `beta` | `0.1.0-beta.1` | Feature complete, stabilization phase |
| `latest` | `latest` | `0.1.0` | Stable release, production ready |

### How It Works

1. GitHub Actions workflow reads `RELEASE_CHANNEL` file
2. Channel determines the npm dist-tag for publishing
3. Channel affects version format (prerelease vs stable)
4. Changing file content changes next release's channel

### Architecture Guidance

**Layer**: Infrastructure (release management)

**Pattern**: Configuration as Code

The file acts as a simple state machine for release progression:
```
alpha → beta → latest
```

To transition channels:
1. Edit `RELEASE_CHANNEL` to new value
2. Commit and push to main
3. Next release uses new channel
